## 📋 核心规则（必须遵守）

### 1. 页面生成规则

**规则**: 所有页面生成后必须附带对应的模拟数据，确保页面可渲染。

**禁止**: 只生成静态结构而不提供数据。

**示例**:
```javascript
// ✅ 正确做法
Page({
  data: {
    services: [
      { id: 1, title: '物命救护', price: 100, image: '/static/service/fish.jpg' },
      { id: 2, title: '法事服务', price: 500, image: '/static/service/ceremony.jpg' }
    ]
  }
})

// ❌ 错误做法
Page({
  data: {
    services: []  // 空数组导致页面无法渲染
  }
})
```

---

### 2. 代码生成前检查规则

**规则**: 生成代码前必须先检查路由配置、数据接口是否存在。

**检查清单**:
- [ ] `app.json` 中是否注册了页面路径
- [ ] 云函数是否已创建并上传
- [ ] API 接口是否已定义
- [ ] 数据结构是否已定义

**不存在时的处理**:
1. 先创建基础接口/数据结构
2. 再编写页面逻辑
3. 最后补充真实数据

**示例流程**:
```
1. 检查 /pages/merit/today 是否在 app.json 中注册
2. 检查云函数 getMeritData 是否存在
3. 如果不存在 → 先创建云函数框架
4. 再生成页面代码
```

---

### 3. 修改后自检规则

**规则**: 每次修改后必须运行项目自检，列出缺失的页面/功能。

**自检命令**:
```bash
# 检查页面文件完整性
find pages -name "*.wxml" | sort

# 检查 app.json 注册
cat app.json | grep pages

# 检查云函数
ls cloudfunctions/
```

**生成修复计划**:
- 列出缺失的页面/功能
- 标注优先级（P0/P1/P2）
- 给出修复时间表

---

### 4. 配置文件保护规则

**规则**: 禁止删除配置文件、路由表。

**修改前必须**:
1. 备份原文件（带时间戳）
2. 记录变更原因
3. 说明影响范围

**示例**:
```bash
# ✅ 正确做法
cp app.json app.json.backup.20260331
# 修改 app.json
# 在 git commit 中说明：添加 merit 页面路由

# ❌ 错误做法
# 直接修改 app.json 不备份
# 直接删除某个页面文件
```

---

### 5. 质量门禁与自动审查规则 ⭐ 新增

**规则**: 每次生成代码前执行质量门禁，生成后自动代码审查，审查不通过自动修复，修复后自动跑自动化测试验证页面可正常渲染。

**执行流程**:
```
1. 生成代码前
   ↓
   运行质量门禁检查
   - bash scripts/quality-gate-checklist.sh
   ↓
2. 生成代码后
   ↓
   自动代码审查
   - quack-code-review 审查
   - bash scripts/code-review-checklist.sh
   ↓
3. 审查结果
   ↓
   通过（>90%）→ 进入测试
   不通过（≤90%）→ 自动修复
   ↓
4. 自动修复
   ↓
   - code-review-fix 修复
   - 重新审查
   ↓
5. 自动化测试
   ↓
   - test-runner 运行测试
   - 验证页面可渲染
   - 测试通过率>95%
   ↓
6. 通过 → 合并
   不通过 → 重新开发
```

**质量门禁标准**:
- [ ] 代码审查通过率 > 90%
- [ ] 测试通过率 > 95%
- [ ] P0/P1问题修复率 100%
- [ ] 页面可渲染（有模拟数据）
- [ ] 路由已注册
- [ ] 文档已更新

**自动修复机制**:
- 小问题（代码风格/格式）→ 自动修复
- 中问题（逻辑错误/缺失数据）→ 提示并修复
- 大问题（架构问题/安全漏洞）→ 打回重做

**测试验证**:
- 运行 test-runner
- 验证页面可渲染
- 检查控制台无 error
- 生成交互测试报告

**示例**：
```javascript
// ✅ 正确做法
Page({
  data: {
    services: [
      { id: 1, title: '物命救护', price: 100, image: '/static/service/fish.jpg' },
      { id: 2, title: '法事服务', price: 500, image: '/static/service/ceremony.jpg' }
    ]
  }
})

// ❌ 错误做法
Page({
  data: {
    services: []  // 空数组导致页面无法渲染
  }
})
```

## 🛠️ 工程思维工具

### A. 静态分析与目录树生成器

**工具**: `tree` 或自定义脚本

**功能**: 在开始写代码前，强制输出当前项目的目录结构。

**使用方式**：
```bash
# 生成项目目录树
tree -L 3 -I 'node_modules|.git'

# 或生成详细列表
find . -type f -name "*.wxml" -o -name "*.wxss" -o -name "*.js" | grep -v node_modules | sort
```

**作用**:
- 确保 AI 对全局文件路径有清晰认知
- 避免路径错误（如 `pages/merit/today` 找不到）
- 快速定位缺失文件

---

### B. API 契约校验器

**工具**: JSON Schema Validator

**功能**: 定义标准 API 返回格式，自动生成 Mock 数据。

**标准 API 格式**：
```json
{
  "code": 200,
  "message": "success",
  "data": {
    "services": [
      {
        "id": "number",
        "title": "string",
        "description": "string",
        "price": "number",
        "image": "string",
        "category": "string"
      }
    ]
  }
}
```

**作用**:
- 发现 `services` 数组为空时，自动生成 Mock 数据
- 补全 cloudfunctions 调用模板
- 确保前后端数据格式一致

---

### C. 组件化重构工具

**工具**: 小程序组件化工具

**功能**: 处理 template、script、style 的分离与集成。

**使用场景**：
- 单文件超过 500 行时，自动建议拆分
- 提取可复用组件（如卡片/按钮/表单）
- 保持代码结构清晰

**组件化标准**：
```
components/
├── service-card/
│   ├── service-card.wxml
│   ├── service-card.wxss
│   ├── service-card.js
│   └── service-card.json
├── order-form/
└── merit-stat/
```

## 📊 质量检查清单

### 提交前检查（必须全部通过）

- [ ] 页面文件完整（wxml/wxss/js/json）
- [ ] app.json 已注册路由
- [ ] 有模拟数据或真实数据
- [ ] 无 console.error 报错
- [ ] 云函数已上传或已创建
- [ ] 配置文件已备份
- [ ] 修改原因已记录

### 代码审查标准

- [ ] 无硬编码路径
- [ ] 无未使用的变量
- [ ] 有适当的错误处理
- [ ] 有必要的注释
- [ ] 符合小程序代码规范

---

## 🚨 违规处理

### 一级违规（严重）
- 删除配置文件不备份
- 生成空数据导致页面崩溃
- 修改路由表不说明原因

**处理**: 立即回滚 + 重新培训 Agent

### 二级违规（一般）
- 未检查路由就生成代码
- 忘记添加模拟数据
- 代码超过 500 行未拆分

**处理**: 警告 + 补充检查清单

### 三级违规（轻微）
- 注释不完整
- 格式不规范

**处理**: 提醒 + 自动修复
