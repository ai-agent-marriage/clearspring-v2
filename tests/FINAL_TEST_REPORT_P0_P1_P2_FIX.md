### 4.2 管理员认证模块 (1 个失败)

**失败用例**: API-AUTH-011: 异常场景 - Token 格式错误

**错误详情**:
```
Expected: "INVALID_TOKEN"
Received: "NOT_FOUND"
```

**根本原因**: 
- Token 格式验证逻辑返回了错误的错误码
- 可能使用了通用的 NOT_FOUND 而非专门的 INVALID_TOKEN

**影响**: 低 (仅影响错误提示准确性，不影响功能)

**修复建议**:
```javascript
// 在 auth 中间件中
if (!isValidTokenFormat(token)) {
  return res.json({ code: 'INVALID_TOKEN', message: 'Token 格式错误' });
}
```

**优先级**: P3 (低优先级)

---

### 4.3 资质审核模块 (1 个失败)

**失败用例**: API-QUAL-009: 异常场景 - 资质不存在

**错误详情**:
```
Expected: "CERTIFICATE_NOT_FOUND"
Received: "QUALIFICATION_NOT_FOUND"
```

**根本原因**: 
- 错误码命名不一致
- 测试用例期望 CERTIFICATE_NOT_FOUND，但 API 返回 QUALIFICATION_NOT_FOUND

**影响**: 低 (仅错误码命名问题，功能正常)

**修复方案** (二选一):
1. 修改 API 错误码：`QUALIFICATION_NOT_FOUND` → `CERTIFICATE_NOT_FOUND`
2. 修改测试用例：更新期望的错误码

**建议**: 统一错误码命名规范，推荐使用 `QUALIFICATION_NOT_FOUND` (更符合业务语义)

**优先级**: P3 (低优先级)

### 4.4 分账配置模块 (2 个失败)

**失败用例 1**: API-PROF-006: 异常场景 - 执行者比例超限

**错误详情**:
```
Expected: "INVALID_EXECUTOR_MIN_RATE"
Received: "INVALID_PLATFORM_RATE"
```

**失败用例 2**: API-PROF-007: 异常场景 - 比例总和超限

**错误详情**:
```
Expected: "INVALID_RATE_SUM"
Received: "INVALID_EXECUTOR_MIN_RATE"
```

**根本原因**: 
- 验证逻辑顺序问题
- API 按固定顺序验证 (平台比例 → 执行者比例 → 总和)，但测试用例期望更精确的错误定位

**验证顺序分析**:
```javascript
// 当前实现
if (platformRate < 0 || platformRate > 1) {
  return { code: 'INVALID_PLATFORM_RATE' }; // 先检查
}
if (executorMinRate < 0 || executorMinRate > 1) {
  return { code: 'INVALID_EXECUTOR_MIN_RATE' }; // 后检查
}
if (platformRate + executorMinRate > 1) {
  return { code: 'INVALID_RATE_SUM' }; // 最后检查
}
```

**影响**: 中 (影响错误提示的精确性，但不影响数据验证)

**修复建议**:
1. 调整验证顺序，优先检查总和
2. 或者修改测试用例，匹配实际验证顺序

**优先级**: P2 (中优先级)
